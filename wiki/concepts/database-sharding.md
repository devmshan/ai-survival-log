# 데이터베이스 샤딩 (Database Sharding)

대규모 데이터를 여러 데이터베이스 서버에 수평 분산하는 기술.

## 개념

```
사용자 데이터
 ├─ 샤드1 (user_id 0-999999)
 ├─ 샤드2 (user_id 1000000-1999999)
 └─ 샤드3 (user_id 2000000+)
```

- 각 샤드(Shard)는 동일한 스키마를 가지지만 다른 데이터 저장
- 데이터베이스의 수평적 규모 확장(Scale Out)

## 샤딩 vs 수직적 확장

| | 수직적 확장 | 샤딩(수평 확장) |
|--|--|--|
| 방법 | 단일 서버 스펙 업 | 여러 서버에 데이터 분산 |
| 한계 | 하드웨어 한계 (Amazon RDS 최대 24TB RAM) | 이론적으로 무제한 |
| 장애 지점 | SPOF 위험 | 샤드별 장애 격리 |
| 비용 | 고성능 서버 단가 높음 | 저사양 서버 여러 대 |

## 수평 샤딩 vs 수직 샤딩

| | 수평 샤딩 (Horizontal) | 수직 샤딩 (Vertical) |
|--|----------------------|-------------------|
| 분리 단위 | 레코드 (행) | 테이블/도메인 |
| 구조 | 같은 스키마, 다른 데이터 | 다른 테이블을 다른 DB로 |
| 언제 | 특정 테이블 데이터가 너무 많을 때 | 도메인이 명확히 나뉠 때 |
| 조인 | 같은 도메인 내 가능 | 서로 다른 DB 간 조인 불가 |

> 보통 "샤딩"이라고 하면 수평 샤딩을 의미

## 수직 샤딩 (Vertical Sharding)

도메인별로 DB를 분리.

```
DB1: 유저 서비스 (users, user_profiles)
DB2: 주문 서비스 (orders, payments)
DB3: 상품 서비스 (products, reviews)
```

**언제 쓰나:**
- 도메인 간 읽기/쓰기 패턴이 다를 때
- 마이크로서비스 아키텍처로 팀이 나뉠 때

**단점 — 조인 문제:**
```sql
-- DB가 나뉘면 SQL JOIN 불가
-- 해결 방법:
-- 1. 애플리케이션 레벨 조인 (네트워크 왕복 2번)
-- 2. 자주 쓰는 데이터 중복 저장
-- 3. 데이터 웨어하우스에서 배치 조인
```

> 버티컬 샤딩은 도메인 간 결합도가 낮을 때 효과적.
> 자주 조인하는 테이블은 같은 DB에 유지할 것.

**대규모 시스템 순서:**
```
버티컬 샤딩(도메인 분리) → 각 도메인 내에서 수평 샤딩
```

## 샤딩 키 (Sharding Key)

데이터를 어떤 샤드에 저장할지 결정하는 컬럼.

```
샤딩 키: user_id
샤드 결정: hash(user_id) % 샤드_수
```

좋은 샤딩 키의 조건:
- 데이터를 균등하게 분산할 수 있어야 함
- 자주 쓰이는 쿼리가 특정 샤드에 집중되지 않아야 함

## 샤딩 방법

| 방법 | 설명 | 장단점 |
|------|------|--------|
| **범위 기반** | user_id 1~1000만 → 샤드1 | 단순하지만 핫스팟 위험 |
| **해시 기반** | hash(user_id) % 샤드수 | 균등 분산, 샤드 추가 시 재배치 필요 |
| **위치 기반** | 한국 유저 → Korea 리전 | 글로벌 서비스 적합, 지역별 불균등 가능 |

> 일반적: 해시 기반 / 글로벌 서비스: 위치 기반 + 해시 기반 조합

## 샤드 장애 복구

샤딩 단독으로는 장애 복구 불가 → 샤딩 + 다중화 조합 필수.

```
샤드 1: 주 DB [A, B] + 부 DB [A, B]
샤드 2: 주 DB [C, D] + 부 DB [C, D]

샤드 2 주 DB 장애 → 부 DB 자동 승격 → 서비스 계속
```

## 샤딩의 문제점

### 1. 재샤딩 (Resharding)
- 한 샤드가 너무 빨리 차거나 데이터 분포 불균형 발생 시 필요
- 샤드 수를 변경하면 기존 데이터를 모두 재배치해야 함
- **해결:** 안정 해시(Consistent Hashing) 기술 사용 (5장에서 상세 다룸)

### 2. 유명인사 문제 (Hotspot Key Problem)
- 특정 샤드에 쿼리가 집중되는 현상
- 예: 유명 연예인 A의 데이터가 한 샤드에만 집중
- **해결:** 유명인 데이터는 별도 샤드 할당 또는 더 세분화

### 3. 조인과 비정규화 (Join and De-normalization)
- 여러 샤드에 걸친 데이터 조인이 어려움
- 샤딩 이후 DB JOIN 사용 불가에 가까워짐
- **해결:** 비정규화(de-normalization)로 단일 테이블에서 데이터 조회 가능하도록 설계

## 언제 샤딩이 필요한가

```
단일 DB 서버 한계 도달
→ 먼저 캐시/읽기전용 복제본 추가
→ 그래도 부족하면 샤딩 고려
```

샤딩은 복잡성이 크게 증가하므로 마지막 수단으로 고려.
