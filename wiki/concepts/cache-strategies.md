# 캐시 전략

캐시(Cache)는 자주 요청되는 데이터를 빠른 임시 저장소에 보관해 응답 속도를 높이는 기법.

## 캐시 계층 (Cache Tier)

```
웹 서버 → 캐시 서버 → (캐시 미스 시) → 데이터베이스
              ↓
          빠른 응답 반환
```

데이터베이스보다 훨씬 빠른 별도의 저장 계층. 데이터베이스 부하를 줄이고 응답 속도를 높인다.

## 캐시 계층 위치

```
클라이언트
  ↓
브라우저 캐시 (내 컴퓨터, 나만 사용)
  ↓
CDN (네트워크 엣지, 모든 유저 공유, 정적 콘텐츠 전용)
  ↓
인메모리 캐시 Redis/Memcached (서버 제어, 동적 데이터)
  ↓
DB (가장 느림)
```

| | 브라우저 캐시 | CDN | 인메모리 캐시 |
|--|-------------|-----|-------------|
| 위치 | 내 컴퓨터 | 네트워크 엣지 | 서버 |
| 공유 여부 | 나만 사용 | 모든 유저 공유 | 서버가 제어 |
| 제어권 | 클라이언트 | 서버 | 서버 |
| 적합한 데이터 | 정적 (JS/CSS) | 정적 (이미지) | 동적 |
| 전략 적용 | 불가 | 불가 | Read/Write-Through 등 |

## Read-Through 캐시 전략

1. 요청 들어옴
2. 캐시에 데이터 있으면(캐시 히트) → 즉시 반환
3. 캐시에 없으면(캐시 미스) → DB 조회 → 캐시에 저장 → 반환

캐시 미스 동시 발생 시 `add` 명령어 사용 → 레이스 컨디션 방지 (두 번째 add는 조용히 실패)

## Write-Through 캐시 전략

쓰기 요청 시 캐시 + DB 동시 업데이트.

```
write 요청 → 캐시 set + DB 업데이트 (동시)
```

- 항상 최신값으로 갱신이 목적 → `set` 명령어 사용
- Read-Through와 함께 쓰면 캐시 미스가 줄어듦
- 단점: DB 쓰기만큼 느림

## Write-Back (Write-Behind) 전략

캐시에만 먼저 쓰고 DB는 나중에 비동기로 저장.

```
write 요청 → 캐시만 업데이트 → 즉시 응답
             (백그라운드에서 DB에 배치 저장)
```

| | Write-Through | Write-Back |
|--|---------------|------------|
| 성능 | DB 쓰기만큼 느림 | 빠름 |
| 데이터 안전 | ✅ 항상 일관 | ❌ 캐시 장애 시 유실 |
| DB 부하 | 높음 | 낮음 |
| 적합한 데이터 | 결제, 주문, 재고 | 조회수, 좋아요, 게임 점수 |

## 캐시 일관성 유지 전략

| 방법 | 일관성 | 복잡도 | 설명 |
|------|--------|--------|------|
| TTL | 약함 | 단순 | 만료 시간만큼 구버전 허용 |
| 명시적 무효화 | 강함 | 중간 | DB 업데이트 후 즉시 delete |
| 이벤트 기반(CDC) | 강함 | 복잡 | 메시지 큐로 자동 무효화 |
| 버전 키 | 강함 | 중간 | 키 변경으로 구버전 자연 소멸 |

> 핵심 판단: "이 데이터가 몇 초 오래돼도 괜찮은가?"

## Memcached API

단순 key-value 캐시. 핵심 명령어:

| 명령어 | 설명 | 언제 쓰나 |
|--------|------|---------|
| `set` | 있으면 덮어씀 | Write-Through |
| `add` | 없을 때만 저장 | Read-Through (레이스 컨디션 방지) |
| `get` | 값 조회 | 읽기 |
| `delete` | 키 삭제 | 캐시 무효화 |
| `incr/decr` | 숫자 증감 | 카운터 |

```python
client.set('user:1', 'minsung', expire=300)  # TTL 300초
client.add('user:1', 'gunseung')  # 이미 있으면 False 반환
client.get_many(['user:1', 'user:2'])  # 다중 조회 (왕복 1번)
```

## 캐시 사용 시 고려사항

### 1. 만료(Expiration) 정책
- 캐시 만료 시간(TTL, Time-To-Live) 설정 필수
- 너무 짧으면: 캐시 효과 없음 (DB 계속 조회)
- 너무 길면: 원본 데이터와 불일치 (stale data)

### 2. 일관성(Consistency)
- 캐시와 DB의 데이터가 항상 같지 않을 수 있음
- 데이터 저장소와 캐시를 동시에 업데이트하는 경우 복잡성 증가
- 분산 환경에서 일관성 유지가 어려움

### 3. 장애 대처 (SPOF 주의)
- 캐시 서버 한 대만 운영하면 단일 장애 지점(SPOF) 위험
- **해결:** 여러 지역(region)에 캐시 서버 분산
- 권장 캐시 메모리 크기: 최소 실제 필요량의 수 배 확보

### 4. 캐시 메모리 크기
- 캐시가 꽉 차면 기존 데이터 방출(eviction) 필요

### 5. 데이터 방출(Eviction) 정책

| 정책 | 설명 | 적합한 경우 |
|------|------|-------------|
| **LRU** (Least Recently Used) | 가장 오래전에 사용된 항목 제거 | 일반적 사용 (가장 흔함) |
| **LFU** (Least Frequently Used) | 사용 빈도가 가장 낮은 항목 제거 | 빈도 기반 접근 패턴 |
| **FIFO** (First In First Out) | 가장 먼저 들어온 항목 제거 | 단순 큐 방식 필요 시 |

## 캐시 대상

캐시에 적합한 데이터:
- 자주 읽히고 드물게 수정되는 데이터
- 계산 비용이 높은 결과 값
- 세션 정보, 인증 토큰 등

캐시에 부적합한 데이터:
- 실시간으로 최신 정보가 필요한 데이터
- 자주 변경되는 데이터 (캐시 무효화 비용이 큼)

## 대표적인 캐시 솔루션

- **Redis:** 인메모리 데이터 구조 저장소, 영속성 지원, 다양한 자료구조
- **Memcached:** 단순 key-value 캐시, 멀티스레드
