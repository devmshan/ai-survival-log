# 시스템 설계 면접 4단계 프레임워크

시스템 설계 면접을 풀어가는 표준 접근법. 정답을 맞히는 게 목적이 아니라, 올바른 과정으로 설계를 진행하는 능력을 보여주는 게 핵심이다.

## 면접의 본질

- 이분법적 정답이 없다. 설계는 열린 문제다
- 면접관은 **협업 능력**, **트레이드오프 추론**, **설계 과정의 논리성**을 평가한다
- 최종 결과물보다 어떻게 생각하고 소통하는지가 더 중요하다

## 4단계 프레임워크

### 1단계: 문제 이해 및 설계 범위 확정

**목적:** 잘못된 가정 위에 설계하지 않기 위해 요구사항을 정확히 파악한다.

절대 면접 시작과 동시에 설계를 시작하지 말 것. 먼저 질문한다.

**필수 확인 항목:**
- 어떤 기능을 만들어야 하는가?
- 사용자 규모 (MAU, DAU)
- 기술 스택 제약 (기존 인프라, 언어 등)
- 확장 속도 요구사항 (지금 당장 글로벌인가, 점진적인가)
- 읽기/쓰기 비율 (읽기 중심 vs 쓰기 중심)

**소요 시간:** 3~10분

### 2단계: 개략적 설계안 제시 및 면접관과 동의 구하기

**목적:** 큰 그림의 청사진을 만들고 방향이 맞는지 면접관과 확인한다.

**포함해야 할 컴포넌트:**
- 클라이언트 (모바일 앱, 웹)
- API 서버
- 데이터 저장소 (RDB, NoSQL)
- 캐시 (Redis 등)
- CDN
- 메시지 큐 (비동기 처리가 필요한 경우)

**진행 방식:**
- 화이트보드(혹은 그림)로 컴포넌트 배치
- 핵심 API 엔드포인트를 간략히 제시
- 면접관에게 "이 방향이 맞나요?"라고 확인

**소요 시간:** 10~15분

### 3단계: 상세 설계

**목적:** 개략적 설계에서 병목이나 핵심 트레이드오프가 있는 부분을 깊게 파고든다.

**집중할 부분:**
- 면접관이 관심을 보이는 컴포넌트
- 성능 병목이 예상되는 부분 (DB 쿼리, 팬아웃, 대역폭)
- 확장성 문제가 발생할 수 있는 지점

**피해야 할 실수:**
- 세부 사항에 너무 집착해서 큰 그림을 놓치기
- 면접관과 소통 없이 혼자만 진행

**소요 시간:** 10~25분

### 4단계: 마무리

**목적:** 설계를 정리하고 놓친 부분을 보완한다.

**다룰 수 있는 내용:**
- 병목 지점과 개선 방법
- 운영 이슈 (지표, 모니터링, 로그, 배포 전략)
- 스케일업 시나리오 (현재 규모 → 미래 규모)
- 미처 다루지 못한 엣지 케이스

**소요 시간:** 3~5분

## 시간 배분 (45분 기준)

| 단계 | 내용 | 시간 |
|------|------|------|
| 1단계 | 요구사항 이해 + 설계 범위 확정 | 3~10분 |
| 2단계 | 개략적 설계 + 면접관 동의 | 10~15분 |
| 3단계 | 상세 설계 | 10~25분 |
| 4단계 | 마무리 | 3~5분 |

## 체크리스트

### 해야 할 것 ✅
- 질문을 통해 모호함을 해소하라
- 이해한 내용을 면접관에게 확인받아라
- 여러 해법을 제시하고 트레이드오프를 설명하라
- 면접관과 지속적으로 소통하면서 진행하라
- 시간이 남으면 설계를 요약하라

### 하지 말아야 할 것 ❌
- 불분명한 상태에서 곧바로 설계 시작
- 면접관과 소통 없이 혼자 진행
- 단일 해법이 있다고 가정
- 설계가 완료됐다고 선언하고 마무리 생략
- 세부 사항에 집착해서 큰 그림을 놓치기
