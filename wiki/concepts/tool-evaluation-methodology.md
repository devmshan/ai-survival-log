# 도구 평가 방법론

도구를 평가할 때 흔히 범하는 실수: "이 도구가 내 프로젝트에 필요한가"를 먼저 판단하기 전에, "이 도구가 무엇을 할 수 있는가"를 정확히 파악하지 않는 것.

이 두 질문은 순서가 있다.

---

## 올바른 평가 순서

```
1. 도구가 실제로 무엇을 하는지 공식 문서로 확인한다
2. 올바른 명령/인터페이스를 찾아서 직접 테스트한다
3. 그 위에서 "내 프로젝트에 지금 필요한가"를 판단한다
```

이 순서가 뒤바뀌면, 도구가 실제로는 할 수 있는 일인데 잘못된 테스트로 "안 된다"고 결론 내리는 오류가 생긴다.

---

## 흔한 실수 패턴

### 1. 서브커맨드 오용

CLI 도구는 서브커맨드마다 역할이 다르다. 잘못된 서브커맨드를 테스트하고 "기능이 없다"고 결론 내리는 경우.

**예시 (Graphify 평가 실패 사례):**

```bash
# 잘못된 테스트
graphify update wiki   # → No code files found
# "마크다운 미지원" 결론

# 올바른 명령 (테스트 안 함)
graphify .             # → 마크다운, 코드, 이미지 전체 처리
```

`update`는 코드 파일 증분 갱신 전용이었다. `graphify .`가 전체 처리 명령이었는데, 이를 테스트하지 않았다.

### 2. 성급한 일반화

몇 번의 실패 후 도구 전체의 능력을 판단하는 것. 테스트한 것은 특정 서브커맨드/경로였는데, 결론은 도구 전체에 적용된다.

### 3. 필요성과 가능성 혼동

"이 도구가 X를 할 수 있는가"와 "이 프로젝트가 X를 필요로 하는가"는 다른 질문이다. 전자 없이 후자를 판단하면, 판단의 근거가 사실이 아닐 수 있다.

---

## 도구 필요성 판단 기준

1단계 (기능 확인) 이후, 2단계 필요성 판단에서 고려할 항목:

| 기준 | 질문 |
|------|------|
| **현재 병목** | 이 도구가 해결하는 문제가 지금 실제로 병목인가? |
| **기존 도구** | 이미 있는 도구로 충분히 해결되고 있지 않은가? |
| **비용** | 의존성, 유지보수, API 비용이 가치를 초과하지 않는가? |
| **원칙 부합** | 프로젝트 운영 원칙과 충돌하지 않는가? |
| **규모** | 현재 규모에서 유효한가, 아니면 미래 규모를 위한 것인가? |

---

## 채택하지 않을 때의 결론 품질

"안 쓴다"는 결론도 근거의 품질이 다르다.

| 낮은 품질 결론 | 높은 품질 결론 |
|----------------|----------------|
| "도구가 못 해서 안 쓴다" (사실 오류 가능) | "도구는 할 수 있지만 지금은 필요 없다" |
| 재검토 기준이 없음 | 재검토 조건 명시 (규모, 필요 기능 등) |
| 문이 닫힘 | 문이 열려 있음 |

높은 품질 결론은 미래의 재평가 시점도 명시한다.
