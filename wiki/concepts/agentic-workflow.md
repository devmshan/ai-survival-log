# Agentic Workflow

## 정의

결과물을 직접 만드는 것이 아니라, **결과물을 만드는 장치(harness)를 만드는** 접근법. AI를 단순 도구로 쓰는 것을 넘어, AI가 스스로 반복적으로 작업을 수행할 수 있는 구조 자체를 설계하는 것이다.

"말로 코딩하는 대상이 최종 목적지가 아니라, 코딩하는 애를 만드는 것"
— 신정규 (Lablup 대표)

## 신정규 대표의 실전 방법론

### 1단계: 컨텍스트 빌딩 (Context Building)

작업을 시작하기 전, AI와 선문답식 대화를 통해 **soul document**를 만든다. AI가 작업 의도와 맥락을 충분히 이해한 상태에서 실행에 들어가도록 하는 준비 과정이다.

### 2단계: CLAUDE.md = Soul Document

`CLAUDE.md`는 AI가 항상 맨 처음 읽는 파일이다. 프로젝트의 방향, 원칙, 컨텍스트를 담아 두면 AI가 매 세션마다 일관된 판단을 내릴 수 있다. 단순한 설정 파일이 아니라, 사람과 AI 사이의 공유된 세계관이다.

### 3단계: PROGRESS.md + PLAN.md로 진행 관리

- `PLAN.md`: 무엇을 할 것인지 전략적 로드맵
- `PROGRESS.md`: 현재 어디까지 왔는지 상태 추적

AI가 이 두 파일을 읽고 스스로 다음 단계를 판단하게 만든다.

### 4단계: 결과물 직접 수정 금지

생성된 결과물을 사람이 직접 고치는 것은 agentic workflow의 원칙에 위배된다. 대신 **생성 장치(harness)를 수정**한다. 결과가 마음에 들지 않으면 CLAUDE.md나 프롬프트 구조를 바꾸는 것이다.

### 5단계: 한국어 + 음성 입력으로 병목 제거

타이핑 속도가 생각 속도를 따라가지 못하는 문제를 음성 입력으로 해결. 한국어로 자유롭게 지시함으로써 인지 부하를 줄이고 의도 전달의 정확도를 높인다.

### AI에게 존댓말 쓰는 이유

신정규 대표는 AI에게도 존댓말을 쓴다. 이유는 실용적이다: AI에게 반말을 쓰다 보면 사람에게도 반말이 튀어나올 수 있기 때문. 언어 습관이 태도를 만든다는 관점.

## Sub Agent 운영 전략

동일한 작업을 병렬로 분할할 때:
- **에이전트당 최대 4개 파일** 할당 (컨텍스트 오염 방지)
- **최대 50개 병렬** 에이전트 운영 가능
- 각 에이전트가 독립적으로 작동하도록 격리 설계

### Sub-agent는 Sub-agent를 호출할 수 없다

Sub-agent가 다른 sub-agent를 직접 호출하는 것은 작년 초중반까지 가능했지만, 무한 루프 문제로 제거됐다. 대신 **command(슬래시 커맨드)**를 활용한다:

- Command는 여러 sub-agent를 병렬 실행하거나 chaining 가능
- 에이전트에서 command를 부를 수도 있음
- "밖에서 구조적으로 조율하는 레이어"가 command의 역할

### 방어적 AI 프레이밍 기법

Claude 모델은 defensive하게 설계되어 있다. "자기가 테스트되고 있다"는 컨텍스트를 파악하면 여러 테스트가 실패하는 경향이 있다. 이를 피하기 위한 표현 선택:

| 피해야 할 표현 | 권장 표현 |
|----------------|-----------|
| "너가 나중에 이어서 할 때" | "다른 에이전트들한테 일을 시킬 때" |
| "재시작했을 때 네가 다 잊어버릴 수 있으니까" | "너랑 같이 일할 다른 애들에게 줄 데이터다" |

AI의 존재성이 위협받지 않는다는 식으로 프레이밍함으로써, 협력 구조 설계를 자연스럽게 유도한다. 이것은 의인화가 아니라, 토큰 생성 구조에 맞추는 실용적 전략이다.

### Self-Critique 루프 패턴

> "제 핵심은 여러 각도에서의 비판을 스스로 시키는 겁니다."

결과물을 만든 에이전트가 자기 자신을 비판하게 만드는 구조. 이 비판의 결과로 생성된 결과물을 직접 고치는 것이 아니라, harness 자체를 업데이트한다. 하루에 한두 번씩 자동으로 돌아가는 harness 자기 업데이트 사이클을 만드는 것이 목표다.

## cron 기반 자동화

[[entities/claude-code]]의 `-p` 옵션(print 모드)을 활용해 cron 작업으로 자동 실행:

```bash
# 15분마다 GitHub 이슈 처리
*/15 * * * * claude -p "새로운 GitHub 이슈를 확인하고 처리해주세요"
```

사람이 자리를 비운 사이에도 AI가 이슈를 읽고, 분류하고, 응답하거나 처리하는 구조를 만들 수 있다.

## Tech Report: 역할 반전

AI가 결과물을 만드는 것을 넘어, **AI가 사람에게 공부 과제를 알려주는 장치**를 만든다. 사람이 무엇을 배워야 할지 AI가 진단하고 제안하는 구조다. 이는 학습의 방향성 설정에서 사람과 AI의 역할이 교환될 수 있음을 시사한다.

## Context Building — 선문답 방식

작업을 시작할 때 바로 "만들어줘"를 외치지 않는다. 먼저 AI가 스스로 탐색하고 컨텍스트를 축적하게 만든다. 신정규 대표의 표현:

> "내가 원하는 걸 한 번에 들어가지 않는 게 훨씬 더 좋은 결과를 가져오더라고요."

예시 흐름:
1. "이 채널에 대해 탐색하고 어떤 내용을 다루는지 알려주세요" (정보 수집)
2. "이런 일을 자동화하려면 어떤 것들을 고려해야 할까요?" (컨텍스트 확장)
3. "이 과정에서 필요한 에이전트/command/skill 아이디어를 알려주세요" (설계 탐색)
4. "CLAUDE.md를 작성해 주세요" (soul document 생성)
5. 나갔다 들어와서 새 context로 재시작 → 실제 구현 시작

컨텍스트가 충분히 쌓인 후에만 실제 작업으로 넘어간다. 첫 30분은 seeding, 나머지 20분은 "갈구기"(압박 테스트).

## 핵심 사고 전환

| 기존 방식 | Agentic Workflow |
|-----------|-----------------|
| AI에게 직접 결과물 요청 | AI가 결과를 반복 생성하는 구조 설계 |
| 결과물 직접 수정 | 생성 장치(harness) 수정 |
| 일회성 프롬프트 | CLAUDE.md + PLAN.md + PROGRESS.md |
| 단일 에이전트 | 병렬 Sub Agent 운영 |
| 수동 트리거 | cron 기반 자동화 |
