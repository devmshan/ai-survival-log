# 시스템 설계 면접 7장 — 분산 시스템을 위한 유일 ID 생성기 설계

분산 환경에서 단순 auto_increment는 통하지 않는다. 이 장은 UUID, 티켓 서버, 스노우플레이크 세 가지 접근의 트레이드오프를 분석하고, 64비트 시간순 정렬 가능 ID를 초당 10,000개 이상 생성하는 설계를 다룬다.

## 핵심 설계 요구사항

- 숫자 전용 64비트 정수
- 시간순 정렬 가능 (타임스탬프 포함)
- 초당 10,000개 이상 생성 가능
- 서버 간 조율 오버헤드 최소화
- 고가용성

## 접근별 트레이드오프

| 방식 | 유일성 | 정렬 | 64비트 | SPOF | 비고 |
|------|--------|------|--------|------|------|
| UUID | ✓ | ✗ | ✗ | ✗ | 128비트, 문자 포함 |
| 티켓 서버 | ✓ | ✓ | ✓ | ✓ | 중앙화 병목 |
| 스노우플레이크 | ✓ | ✓ | ✓ | ✗ | 권장 |

## 트위터 스노우플레이크 구조

```
[1비트 사인] [41비트 타임스탬프] [5비트 DC ID] [5비트 서버 ID] [12비트 일련번호]
```

- 타임스탬프(41비트): 기원 이후 밀리초, 약 69년치
- 일련번호(12비트): 같은 밀리초 내 최대 4096개, 다음 밀리초 초기화

타임스탬프가 상위 비트 → 정렬 자동 보장.

## 면접 포인트

- "시계 동기화는 어떻게?" → NTP 사용, 시계 역행 시 대기 또는 예외 처리
- "비트 구성을 바꿀 수 있나?" → 예: 동시성 낮으면 일련번호 비트 줄이고 타임스탬프 늘려 수명 연장
- "기원(epoch)은 어떻게 설정?" → 트위터 스노우플레이크는 2010-11-04 01:42:54 UTC를 기원으로 사용
