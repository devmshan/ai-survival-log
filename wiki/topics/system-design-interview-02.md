# 대규모 시스템 설계 스터디 02 — 개략적인 규모 추정

> 가상 면접 사례로 배우는 대규모 시스템 설계 기초 (Alex Xu) 2장 스터디 기록.
> 질문하고 답하면서 체화한 내용을 그대로 옮긴다.

---

## 개략적 추정이란

면접관이 "트위터 같은 서비스의 서버가 몇 대 필요할까요?"라고 물으면, 당신은 계산기도 없이 5분 안에 답해야 한다.

이걸 **Back-of-the-Envelope Estimation**이라고 부른다. 냅킨 뒷면에 대략적인 계산을 끄적이는 것처럼, 정밀하지 않아도 올바른 규모(order of magnitude)의 감각을 갖추는 훈련이다.

2장은 이 감각을 만드는 세 가지 도구를 소개한다.

---

## 도구 1: 2의 제곱수 — 단위 감각

데이터 볼륨을 다룰 때 모든 계산의 출발점은 단위다.

| 단위 | 규모 |
|------|------|
| 1KB | 10³ bytes |
| 1MB | 10⁶ bytes |
| 1GB | 10⁹ bytes |
| 1TB | 10¹² bytes |
| 1PB | 10¹⁵ bytes |

실전 감각으로 변환하면:
- 트윗 텍스트 1개 ≈ 200B
- 이미지 1장 ≈ 300KB~1MB
- HD 영상 1분 ≈ 50~150MB

---

## 도구 2: 응답지연 기준값 — 느린 것을 피하는 감각

Jeff Dean이 정리한 이 숫자들은 "무엇을 피해야 하는가"를 말해준다.

| 연산 | 지연 |
|------|------|
| L1 캐시 참조 | 0.5ns |
| L2 캐시 참조 | 7ns |
| 주 메모리 참조 | 100ns |
| SSD 임의 읽기 | 150µs |
| HDD 탐색 | 10ms |
| 같은 DC 패킷 왕복 | 0.5ms |
| 대륙 간 패킷 왕복 | 150ms |

이 숫자에서 나오는 교훈:

**메모리는 빠르고 디스크는 느리다.** L1 캐시(0.5ns)에서 HDD 탐색(10ms)까지 2,000만 배 차이다. 시스템 설계에서 "디스크 읽기를 줄이자"는 말이 얼마나 중요한지 이 숫자가 설명한다.

**데이터센터 내부는 빠르고 대륙 간은 느리다.** 같은 DC(0.5ms)와 캘리포니아↔네덜란드(150ms)는 300배 차이다. 글로벌 서비스라면 데이터를 사용자 가까운 곳에 두는 게(CDN, 지역 분산 DC) 응답속도에 결정적이다.

---

## 도구 3: 가용성 수치 — 나인을 세는 감각

가용성(Availability)은 시스템이 정상 작동하는 시간의 비율이다.

| 가용률 | 별칭 | 1년 장애 허용 |
|--------|------|---------------|
| 99% | two nines | 3.65일 |
| 99.9% | three nines | 8.77시간 |
| 99.99% | four nines | 52.6분 |
| 99.999% | five nines | 5.26분 |

99%와 99.999%는 숫자로는 0.099% 차이지만, 허용 장애 시간은 연간 3.65일 vs 5.26분 — 1,000배 차이다.

아마존/구글/마이크로소프트는 99.9% 이상 SLA(Service Level Agreement)를 공식 약속한다. 계약에 명시된 가용성을 못 지키면 크레딧을 돌려줘야 한다.

"우리 시스템 가용성이 얼마야?"라는 질문은 결국 "연간 몇 분 죽어도 되는가?"로 바꿔 읽을 수 있다.

---

## 실전 예제: 트위터 QPS와 저장소 추정

면접에서 이 세 도구를 어떻게 조합하는지 보여주는 예제다.

### 가정을 먼저 명시한다

- MAU(월간 활성 사용자): 3억 명
- DAU = MAU × 50% = **1.5억 명**
- 하루 트윗: 사용자당 평균 2건
- 트윗의 10%에 미디어 첨부 (평균 1MB)
- 데이터 보관 기간: 5년

### QPS 계산

```
하루 트윗 수 = 1.5억 × 2 = 3억 건
QPS = 3억 / 86,400초 ≈ 3,500
Peak QPS = 3,500 × 2 ≈ 7,000
```

하루에 3억 개 트윗이 올라오고, 초당 약 3,500개가 처리된다. 피크 타임엔 7,000개.

### 저장소 계산

```
트윗 텍스트 크기: tweet_id(64B) + 텍스트(140B) = 204B
텍스트/일: 3억 × 204B ≈ 61.2GB
미디어/일: 3억 × 10% × 1MB = 30TB

5년 총 저장소: (30TB + 0.06TB) × 365 × 5 ≈ 54.75PB
```

텍스트(61.2GB)와 미디어(30TB)를 비교해보면, 텍스트는 미디어의 0.2% 수준이다. **미디어가 저장소를 지배한다.** 이미지/영상을 다루는 시스템은 스토리지 설계가 핵심이다.

---

## 추정할 때 지켜야 할 습관

1. **가정을 먼저 명시한다.** "DAU를 1.5억으로 가정하겠습니다."
2. **근사치를 과감하게 사용한다.** "98,673 / 9.1 ≈ 100,000 / 10 = 10,000"
3. **단위를 항상 붙인다.** "5"가 아니라 "5MB"
4. **자주 구하는 값을 알아둔다:** QPS, Peak QPS, 저장소, 캐시 크기, 서버 수
