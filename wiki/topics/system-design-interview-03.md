# 대규모 시스템 설계 스터디 03 — 면접 공략법

> 가상 면접 사례로 배우는 대규모 시스템 설계 기초 (Alex Xu) 3장 스터디 기록.
> 질문하고 답하면서 체화한 내용을 그대로 옮긴다.

---

## 시스템 설계 면접은 왜 다른가

알고리즘 면접은 정답이 있다. 정렬 알고리즘을 맞히면 맞힌 거고, 틀리면 틀린 거다.

시스템 설계 면접은 다르다. "트위터를 설계하세요"라는 질문에 정해진 정답이 없다. 어떤 설계도 완벽하지 않다. 트레이드오프가 항상 존재한다.

그래서 면접관이 보는 건 **최종 설계물**이 아니다. 어떻게 생각하고, 어떻게 소통하고, 어떻게 트레이드오프를 따지는지 — **그 과정**이다.

3장은 이 과정을 4단계 프레임워크로 체계화한다.

---

## 1단계: 문제 이해 및 설계 범위 확정

면접이 시작되면 가장 먼저 드는 충동은 "일단 뭔가 그려야지"다. 이걸 참아야 한다.

잘못된 가정 위에 세운 설계는 아무리 정교해도 쓸모없다. 먼저 질문한다.

**물어봐야 할 것들:**

기능 관련:
- "구체적으로 어떤 기능을 만들어야 하나요?"
- "모바일과 웹 둘 다 지원해야 하나요?"
- "이미지, 동영상도 올릴 수 있나요?"

규모 관련:
- "사용자 수는 몇 명입니까?"
- "DAU는 얼마나 되나요?"
- "트래픽 규모가 어느 정도인가요?"

기술 제약 관련:
- "주로 어떤 기술 스택을 사용합니까?"
- "기존 인프라를 활용해야 하나요?"

이 질문들로 **설계 범위**를 확정한다. 뉴스 피드를 설계하더라도, "좋아요 기능 포함 여부", "피드 정렬 기준", "팔로우 한도" 같은 세부 결정에 따라 설계가 완전히 달라진다.

시간: **3~10분**

---

## 2단계: 개략적 설계안 제시 및 동의 구하기

범위가 확정됐으면 큰 그림의 청사진을 그린다. 정밀하지 않아도 된다. 방향이 맞는지 확인하는 게 목적이다.

**포함할 컴포넌트:**

```
클라이언트 (앱/웹)
  ↓
API 서버
  ↓
캐시(Redis) / 데이터 저장소(RDB, NoSQL) / CDN / 메시지 큐
```

그린 다음 면접관에게 묻는다: "이 방향이 맞나요?"

이 단계에서 핵심 API도 간략히 제시한다.

```
# 뉴스 피드 예시
POST /v1/me/feed          # 포스트 발행
GET  /v1/me/feed          # 피드 조회
```

단순히 그림만 그리는 게 아니라, **설계 결정을 면접관과 함께 내리는** 과정이다.

시간: **10~15분**

---

## 3단계: 상세 설계

개략적 설계에서 병목이나 트레이드오프가 있는 부분을 골라 깊게 파고든다. 모든 컴포넌트를 다 파고들 시간은 없으니 **우선순위를 정해야** 한다.

면접관이 관심 보이는 컴포넌트, 성능이 가장 크게 영향받는 지점에 집중한다.

**뉴스 피드 상세 설계 예시:**

피드 발행 흐름:
```
사용자 → 포스트 서비스 → DB 저장
                       → 팬아웃 서비스 → 친구들 피드 캐시에 추가
                                       → 알림 서비스 → 푸시 발송
```

피드 조회 흐름:
```
사용자 → 피드 서비스 → 피드 캐시 조회
                    → (캐시 미스면) DB 조회 후 캐시 저장
                    → 미디어 콘텐츠는 CDN에서 반환
```

여기서 나올 수 있는 핵심 트레이드오프:
- **팬아웃 방식**: 쓰기 시점(Push) vs 읽기 시점(Pull)
  - Push: 팔로워가 적으면 빠름. 팔로워 1억 명인 셀럽이라면 쓰기 폭발
  - Pull: 읽을 때 집계. 읽기 느림. 셀럽 계정엔 유리
  - 현실: 일반 사용자는 Push, 대형 계정은 Pull — 혼합 방식

시간: **10~25분**

---

## 4단계: 마무리

면접 후반, 면접관이 추가 질문을 던진다. 몇 가지 방향이 있다.

**운영 이야기를 꺼낼 수 있다:**
- 어떤 지표를 모니터링할 것인가?
- 로그는 어떻게 수집하나?
- 배포는 어떻게 할 것인가?

**스케일업 시나리오:**
- "지금은 DAU 1만 명인데, 1년 뒤 100만 명이 되면?"
- 어느 컴포넌트부터 확장이 필요한지 짚어준다

**엣지 케이스:**
- 셀럽 계정처럼 팔로워가 극단적으로 많은 경우
- 트래픽 폭발 시나리오 (스포츠 경기 실시간 등)

시간: **3~5분**

---

## 정리: 해야 할 것과 하지 말아야 할 것

### 해야 할 것
- 질문을 통해 모호함을 해소한다
- 이해한 내용을 면접관에게 확인받는다
- 여러 해법을 제시하고 트레이드오프를 설명한다
- 면접관과 계속 소통하면서 진행한다
- 시간이 남으면 설계를 요약한다

### 하지 말아야 할 것
- 모호한 상태에서 곧바로 설계 시작 ❌
- 면접관과 소통 없이 혼자만 진행 ❌
- 단일 해법이 있다고 가정 ❌
- 세부 사항에 집착해서 큰 그림 놓치기 ❌
- 설계가 완료됐다고 선언하고 마무리 생략 ❌
