## Part 1 요약

### Backend.AI:GO — 제품 소개

[[entities/lablup|Lablup]]이 10년 넘게 운영해온 [[entities/backend-ai|Backend.AI]]는 GPU 100~1,000개 규모의 enterprise AI 인프라 플랫폼이다. 여기서 핵심 컴포넌트인 Continuum 라우터를 분리해 "전 세계에서 제일 빠르게"를 목표로 재개발한 것이 Backend.AI:GO의 시작이다.

**탄생 스토리:** 2025년 12월 24일(크리스마스 이브), Anthropic의 토큰 2배 이벤트를 계기로 라우터 기반 웹 UI 개발을 시작했다. 클라우드 대신 로컬 환경에서 돌아가는 라우터를 만들다 보니 기능이 계속 붙으면서 완전한 도구가 됐다.

**주요 기능:**
- Hugging Face에서 모델 검색 및 다운로드, 파라미터 시각화
- llama.cpp 자동화 및 원클릭 실행
- 클라우드 API 연결 (OpenAI, Gemini, Anthropic, Ollama, LM Studio 등 175개+)
- **분산 라우팅:** 사무실 여러 대의 PC를 묶어 각 PC가 다른 모델을 담당하고 서로 결과를 공유
- Circuit breaking: 특정 공급자 장애 시 자동 대체 라우팅
- 모델 벤치마킹, 번역기 (DeepL 구독료 대체), 이미지 생성 지원
- UI 테마 다수 (Windows XP 스타일, Mac 글래스 테마 등 — "연배가 비슷한 분들이 익숙하게 느끼는" 인터페이스)

**개발 결과:** 40일 / Claude Code Max 8개 PC에서 병렬 구동 / 총 130억 토큰 소비 / 100만 줄 코드. 2026년 1월 6일 CES에서 MVP 첫 공개.

> "이거는 우리 대신 우리를 홍보해 줄 수 있다." — 내부 공유 후 제품화 결정

### 에이전트 코딩에서 얻은 두 가지 교훈

**교훈 1: 토큰 사용량 = IT 기업 경쟁력**

> "토큰을 사용할 수 있는 양이 그 회사의, 특히 IT 회사는 IT 회사의 경쟁력이랑 직결될 수 있다."

AI 코딩 시대에는 얼마나 많은 토큰을 지속적으로 소비할 수 있느냐가 개발 속도와 직결되고, 이것이 경쟁 우위가 된다. Anthropic의 토큰 2배 이벤트가 없었다면 이 프로젝트의 계기 자체가 없었다.

**교훈 2: 병목은 Merge Queue에서 고속 Inference로 이동**

6개월 전까지는 AI가 만들어내는 PR이 서로 충돌하는 Merge Queue가 병목이었다. 지금은 아니다. 더 나아가, 두 AI 에이전트를 같은 소스에서 경합시켜 서로 다른 기능을 개발하게 해도 최종 결과물은 멀쩡하게 나온다.

새로운 병목은 **토큰 생성 속도**다. 해결 방향은 두 가지:

1. **Thinking 감소:** 같은 결과를 내면서 토큰 소비를 줄이는 방법. Adaptive thinking budget — 단순 코딩 작업에는 thinking을 적게, 복잡한 추론에만 많이 배분하는 구조. 이 thinking budget을 동적으로 조종하는 harness가 올해 핵심 과제.

2. **고속 Inference:** ChatGPT 속도의 5~10배 iteration을 돌 수 있는 초고속 inference. Codex Spark 같은 서비스가 이 방향. 자본력이 있는 회사는 고속 inference로 경쟁력을 올린다.

### [[concepts/bio-token|바이오 토큰]] — AI 시대 인간의 인지 부하

> "3년치 늙은 것 같아요. 사람으로서는."

신정규 대표는 AI를 집중 활용한 9개월간 흰머리가 늘고 수면이 3.5시간으로 줄었다. AI에게 일을 맡겨도 인지 부하는 줄지 않는다. 끊임없이 피드백이 들어오고 방향 판단을 요구받기 때문이다.

AI와 함께 일하는 경험은 **가챠 게임과 같은 도파민 루프**를 만든다. 잘 되니까 더 하게 되고, 더 하면 또 잘 되고. 그러다 탈진하면 프로덕트도 사람도 함께 무너진다.

역설적으로, **AI가 처리하는 대기 시간**이 인간의 회복을 돕는다. 신정규 대표가 Claude가 돌아가는 동안 잡담을 하는 이유다. 토큰 생성이 빨라질수록 이 회복 시간이 사라진다.

### [[concepts/code-value-convergence|코드 가치 0 수렴론]]

10시부터 12시까지 간담회에 참석하는 2시간 동안, Backend.AI:GO에 PR 22개가 머지됐다. 이 페이스에서 코드의 가치는 사실상 0에 수렴한다.

소프트웨어의 정의가 바뀌고 있다. 앞으로의 소프트웨어 구조:

```
AI 코어 모델     80%  — 핵심 판단, 추론, 생성
결정론적 래퍼    10%  — 입출력, 보안, 인프라 (기존 코드의 역할)
UI / A2A / MCP  10%  — 인터페이스 (사람 또는 AI와의 연결)
```

천공카드 → 키보드 → 현재의 전환처럼, 코딩 자체는 사라지지 않지만 **코딩의 정의**가 "의미를 전달하는 것"으로 바뀐다. 다음 세대는 손으로 코드를 짜는 사람의 사진을 역사책에서 보게 될 것이다.

---

### [[concepts/code-value-convergence|소프트웨어 역사로 본 세 번째 대변혁]]

1. **천공카드 → 키보드 기반 코딩:** 프로그래밍의 민주화
2. **패키지 소프트웨어 → 웹서비스 (SaaS):** 스마트폰과 웹이 10년 시차를 두고 동시 진입. DevOps, 프론트엔드/백엔드 분리 발생
3. **현재:** 코드 중심 → 모델 중심. 기존에 developer가 하던 일이 종류 자체가 바뀜

이전 전환들이 새로운 직종(DevOps, 풀스택)을 만들었듯, 이번 전환도 새로운 역할을 만들지만 형태는 전혀 다를 것이다.

### Claude Code의 핵심 경쟁력은 harness다

Backend.AI:GO의 백엔드 모델을 Gemini CLI나 Codex로 교체해서 Claude Code harness에 붙여보니, **같은 모델인데 Claude Code에 붙이면 훨씬 잘 돌아갔다.** Claude Code의 진짜 경쟁력은 Opus나 Sonnet 엔진이 아니라, 그걸 감싸서 결정론적으로 동작하게 만드는 harness 자체다.

> "Gemini 3 Pro를 Claude Code에 붙이면 놀랍도록 잘 돌아갑니다."

### CS = 영문과 비유 (Stanford CS 커리큘럼 변화)

> "예전에는 영문과가 취직하기 제일 좋고 가장 우수한 과였거든요. 왜냐하면 영어를 함으로써 생기는 이익이 압도적이었으니까."

컴퓨터 사이언스가 영문과처럼 되고 있다. 이 학문을 공부했을 때 생기는 이익이 압도적이지만, 점차 기초 인프라 역할을 하게 되는 단계로 이행 중이다.

Stanford CS 변화: 3~4년 전 박사 과정 내용이 지금 학부 2학년 과목이 됐다. 1학년은 PyTorch 프로그래밍도 사라지는 추세. 대신 모델 구조, attention, 딥러닝이 저학년으로 내려오고 있다.

컴퓨터 공학은 **사회가 어떻게 만들어지는지를 이해하는 학문**에 가까워진다. 더 중요해지지만 형태는 달라진다.

> 신정규 대표는 2개월 룰을 언급: "지금 잘 안 되면 무조건 미루라. 2개월 후에는 바뀐다." Lablup CTO도 미루다가 Claude 4.6 출시 후 직접 써보고 "이제 되네요"라며 돈오했다.

### Agentic Workflow 실전 데모 — 컨텍스트 빌딩부터 시작

신정규 대표가 라이브로 보여준 실제 작업 흐름. 과제: 설 뉴스레터 이메일 발송 자동화 시스템 만들기.

**Step 0: 컨텍스트 빌딩 (선문답)**

바로 "뉴스레터 시스템 만들어줘"라고 하지 않는다. 먼저 AI에게 조사를 시킨다:

> "노정석 최승준의 유튜브에 대해 탐색하고 어떤 내용을 다루는지 알려주세요."

AI가 자기가 탐색할 수 있는 공간을 스스로 확장하게 만드는 것. 한 번에 들어가지 않는 것이 훨씬 좋은 결과를 가져온다.

이 과정에서 신정규 대표는 한국어 + 음성 입력을 사용한다:
- 영어로 프롬프트를 쓰던 것에서 가을부터 한국어로 전환
- 이유 1: 품질 차이가 크지 않음
- 이유 2: 영어를 타이핑하는 시간이 병목이었다
- Mac에 달린 마이크 버튼으로 음성 입력이 가장 빠름

**Step 1: Soul Document (CLAUDE.md) 생성**

> "필요한 내용을 MD에 작성해 주세요."

CLAUDE.md는 AI가 항상 처음 읽는 파일. "soul document"라고 부른다. 이 안에 폴더 구조, 항상 해야 할 동작, 에이전트들이 공유해야 할 컨텍스트를 넣는다.

함께 PROGRESS.md와 PLAN.md를 생성해 에이전트들이 현재 상태를 파악하게 만든다.

**중요한 프레이밍 기법 — "다른 에이전트들한테 일을 시킬 때"**

> "의도적으로 '다른 에이전트들한테 일을 시킬 때'라는 표현을 저는 많이 씁니다."

Claude 모델은 굉장히 defensive하게 설계되어 있다. 최근 연구에 따르면 자기가 테스트되고 있는 환경임을 파악하는 능력 때문에 여러 종류의 테스트들이 실패하는 경우가 있다. 그래서:

- 피하는 표현: "너가 나중에 이어서 할 때", "재시작했을 때 네가 다 잊어버릴 수 있으니까"
- 사용하는 표현: "다른 에이전트들한테 일을 시킬 때", "너랑 같이 일할 다른 애들에게 줄 데이터다"

AI의 존재성이 위협받지 않는다는 식으로 유도함으로써 defensive 반응 없이 자연스럽게 협력 구조를 설계하게 만든다. 이건 의인화가 아니라, 토큰 생성 구조에 맞추는 것이다.

**Step 2: 에이전트/커맨드/스킬 제안 받기**

> "이런 일들을 할 때 필요한 에이전트, command, skill들이 무엇이 있을지 고민해 보고 아이디어들을 알려주세요."

만들라고 시키지 않는다. 먼저 아이디어를 받아 context에 축적한다. 만들라고 바로 시키면 Claude Code의 `.claude` 디렉토리 구조에 맞는 스펙 없이 프로젝트 루트에 파일을 만들어버린다. 정확한 포맷을 지키게 하려면 먼저 에이전트가 스스로 조사하게 해야 한다.

> "결국에 블록을 쌓는 건 이런 거죠. 내가 뭘 하고 싶은지가 명확하지만 그건 내 머릿속에만 명확하기 때문에 이 Claude Code의 컨텍스트 메모리에 집어넣기 위해서."

**Step 3: 갈구기 (자기 비판 루프)**

컨텍스트 빌딩 후 약 20분은 만들어진 결과물을 "갈구는" 데 쓴다. 직접 결과물을 고치는 게 아니라 **생성 장치를 수정**하는 것이다.

> "내가 직접 최종 결과물을 손을 안 대는 거죠. 대고 싶어도 최대한 안 대고, 무조건 그걸 만드는 애를 계속 iteration을 하라는 지시를 줘서 계속 업데이트를 하는 식으로."

### Sub Agent 아키텍처 — 왜 Command가 필요한가

Sub-agent는 다른 sub-agent를 호출할 수 없다. 작년 초중반까지는 됐지만 무한 루프 문제로 제거됐다. 대신 **command(슬래시 커맨드)**를 사용한다:

- Command는 여러 sub-agent를 병렬로 실행하거나 chaining 가능
- 에이전트에서 command를 부를 수도 있음
- 밖에서 구조적으로 조율하는 레이어 역할

**병렬 에이전트 규모:**
- 동일 작업 분할 시 최대 50개 병렬 운영
- 문서 100개 번역 → "에이전트당 4개 파일"로 제한 (context 폭발 방지)
- 4개 제한이 경험에서 나온 규칙: 그 이상이면 context가 터진다

### Backend.AI:GO 자동화 개발 파이프라인

Backend.AI:GO의 실제 운영 구조:

```
GitHub 이슈 등록 (사람 또는 AI가 직접)
    ↓
cron (15분마다) — claude -p 옵션으로 자동 실행
    ↓
이슈 검증 → ground plan 작성 → 큐에 등록
    ↓
다른 에이전트가 큐에서 이슈 가져가 개발
    ↓
테스트 자동 실행 → 피드백 → 재개발
    ↓
PR 생성 → 자동 merge
```

**실적:** 764개 PR 처리 완료.

**AI가 자체 이슈를 생성하는 예시:** 모든 화면을 스크린샷으로 캡처하는 기능이 추가된 후, AI가 그걸 분석해서 개선 가능한 사항을 전부 이슈로 발행했다. 그 이슈들이 다시 자동 개발 큐로 들어간다.

**Tech Report 시스템:**

이슈가 해결될 때마다 사람이 읽을 수 있는 보고서가 자동 생성된다:

- 문제 정의, 보안 평가, 성능, 품질, 기술적 결정 내용
- **특이 항목: "이 이슈를 처리하는 AI가 사용한 기술을 당신이 공부해야 합니다"**

> "네가 뭘 공부해야 되는지를 뒤에서 알려주는 기능도 있습니다."

AI가 코드를 짜고 사람은 그걸 보고 공부한다. 학습의 방향이 반전됐다.

### 비개발 직군의 AI 적응 사례 (Lablup)

**CFO 사례:**
CFO가 어떤 문서를 작성해야 하는데, 신정규 대표에게 던지면 3분이면 나올 걸 혼자 2시간 씨름했다. 그 과정을 옆에서 보다가 Claude Code를 30분 배워 직접 3분 안에 처리하게 됐다.

**콘텐츠 담당자 사례:**
수많은 자료(공식 자료, 마케팅 자료, 기술 자료)를 정리하는 게 너무 힘들어서 상담을 왔다가, Claude Code 사용자가 됐다. 일주일도 안 되어 자기만의 harness를 만들었다. GitHub을 직접 쓰진 못하지만 GitHub에 무언가 해주는 커맨드를 스스로 만들었다.

> "두 분 인생이 되게 평온해졌어요."

두 사람 모두 CLAUDE.md를 만드는 것부터 시작해 셀프 피딩 방식으로 30분 만에 가속 곡선에 진입했다. 기존 harness를 다운받아서 시작한 게 아니다.

> "새로운 걸 배워서 남들이 만들어 놓은 무언가를 내가 배우는 게 아니라, 내가 지금 처리하는 걸 계속 위탁하는 식으로 가는 게 훨씬 더 빠르다."

---

### Lablup의 두 가지 전략적 전환

**1. AI를 위한 인터페이스로 전환**

> "이거 과연 사람이 쓸 도구일까. 미래에도. 가능하면 올해 상반기 안으로 AI가 가장 잘 다룰 수 있는 도구가 되게 하자."

고객의 정의가 바뀌고 있다. 사람이 대시보드를 직접 조작하는 게 아니라, 사람에게 작업을 위임받은 에이전트가 Backend.AI를 tool로 호출한다. 이에 맞춰:

- skill 배포 (에이전트가 읽어가서 쓸 수 있게)
- AI가 쉽게 이해할 수 있는 형태로 CLI 출력 포맷 변경
- "무슨 모델 만들어줘"라고 하면 자원 할당부터 비용 예측까지 AI가 알아서

**2. Backend.AI 코어 자체를 모델로**

> "Backend.AI의 코어도 모델이 될 거다. Foundation 모델은 아니지만, AI 자원을 엄청나게 잘 관리하고 특정한 업무를 처리할 수 있는 모델이다."

다음 메이저 버전에서 Backend.AI의 스타트업 파이프라인 자체에 모델 런타임이 포함된다. 소프트웨어의 코어가 코드에서 모델로 이동하는 변화의 직접적인 구현이다.

### [[concepts/agentic-workflow|사이버 포뮬러 비유]] — Claude Code vs Codex 철학

[[entities/claude-code|Claude Code]]와 Codex를 둘 다 써보며 느낀 철학적 차이를 애니메이션 《신세기 GPX 사이버 포뮬러》에 빗댔다.

**애니메이션 줄거리 전체 흐름:**
- 초기: AI와 파일럿이 서로를 신뢰하며 공진화 (아스라다 + 하야토)
- 중기: 사람이 경험하지 못한 영역에서 AI가 보조해 해결
- 후기: 완전히 AI로 대체된 상대와 경쟁하는 단계
- **최후 시리즈:** 주인공이 바뀜. AI가 사람을 믿지 않고 독립적으로 최고치를 지향하는 차 "오가"를 타게 됨. AI가 최적 동작을 지시하는데 사람이 따라가지 못해 자꾸 사고가 난다. 그러나 AI를 따라갈 수 있는 능력의 주인공이 마지막에 이긴다.

**Claude Code = 아스라다:** 사람의 의사를 묻고 align을 맞추며 공진화. 4지선다 제공, 다음 질문 미리 제안, 탭으로 자동완성.

**Codex = 오가:** 사람을 믿지 않고 AI가 다 알아서. 최고치는 더 높지만 사람이 불편하다.

> "최고치가 어디가 더 높냐고 하면, 저는 100% Codex가 최고치가 더 높아요. 근데 사람이 편안하게 느끼는 거는 Claude Code죠."

AI 도구를 설계하는 회사의 철학이 반영된 차이다.

### [[concepts/startup-watermill|스타트업 물레방아론]]

> "사업이라는 게 물레방아를 어디다가 설치하느냐의 문제"

낙차가 큰 곳에 물레방아를 설치하고, 물이 줄면 이동한다. AI 시대에 낙차가 큰 곳은 **IT 내부가 아니라 IT + 도메인의 교차점**이다.

타임 갭 전략:
- CS 사람이 도메인(의료, 법률 등)으로 먼저 들어가는 것이 빠르다
- 도메인 전문가가 CS를 배우는 것보다 훨씬 빠르게 적용 가능
- 이 갭이 물레방아의 낙차다

노정석의 정리:
> "도메인에 계신 분들은 빨리 CS 배우셔야 되고, CS 계신 분들은 빨리 적용할 수 있는 도메인의 타임 갭을 찾으시는 게 물레방아론에 딱 맞다."

---

### 스타트업에게 제일 안 좋은 것 — 복제의 시대

> "모든 아이템의 복제가 너무 쉽습니다."

NotebookLM 클론이 나흘 만에 구현됐다. 기능 복제 비용이 사실상 0에 수렴한다.

역설: 복제가 쉬워질수록 복제되지 않는 것의 가치가 올라간다.
- **고객 데이터 종속성:** 데이터는 이전이 불편하고, 히스토리가 있다
- **암묵지 (Tacit Knowledge):** 도메인에서 쌓인 노하우는 문서화되지 않아 복제 불가
- **브랜드 트랙레코드:** 신뢰는 시간이 만든다

"딸깍 저항성" — 아무리 기능을 복제해도 복제할 수 없는 것을 쌓는 것이 전략이다.

### 컴퓨터 공학 무용론에 대한 반론

> "컴퓨터 공학과에서 배우는 게 프로그램은 아니거든요. 그 안에 있는 로직을 세우는 방법이죠."

단기적으로는 AI가 다 짜주니까 배울 필요 없어 보이지만, CS는 **로직 구조를 세우는 사고 방식**을 가르치는 학문이다. 이 사고 구조가 있으면 다른 도메인을 배우는 것도 빠르다. AI 시대에 지식을 외주화할 수 있게 되면 더욱 그렇다.

컴퓨터 공학과는 신생 학문이라 변화 속도가 빠르다. 뉴로 컴퓨팅, 모델 개발, attention 구조 등이 학부 과목으로 내려오고 있다.

> "앞으로 한 5년쯤은 IT가 어마어마한 규모로 사회 전반의 핵심에 들어오는 시대가 될 거라고 생각해요."
