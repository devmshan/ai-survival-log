# 대규모 시스템 설계 기초 2장 — 개략적인 규모 추정

- **원본:** raw/books/스크린샷 2026-04-15 오후 1.40.07~17.png (3장)
- **책:** 가상 면접 사례로 배우는 대규모 시스템 설계 기초 (Alex Xu)

## 핵심 요약

면접에서 "이 시스템의 QPS가 얼마야?", "저장소는 얼마나 필요해?" 같은 질문에 빠르게 답하기 위한 개략적 추정(Back-of-the-envelope estimation) 훈련. 정밀한 계산이 아니라 올바른 규모의 감각을 갖추는 게 목표다.

## 주요 포인트

### 1. 2의 제곱수 — 단위 감각

데이터 볼륨을 다룰 때는 2의 제곱수로 환산한다.

| 단위 | 근사값 | 영어 명칭 |
|------|--------|-----------|
| 1KB | 10³ bytes | 1 Thousand |
| 1MB | 10⁶ bytes | 1 Million |
| 1GB | 10⁹ bytes | 1 Billion |
| 1TB | 10¹² bytes | 1 Trillion |
| 1PB | 10¹⁵ bytes | 1 Quadrillion |

### 2. 응답지연(Latency) 기준값

프로그래머가 알아야 할 연산별 대략적 응답시간 (Jeff Dean 원본 기반):

| 연산 | 지연 시간 |
|------|-----------|
| L1 캐시 참조 | 0.5ns |
| 분기 예측 오류 | 5ns |
| L2 캐시 참조 | 7ns |
| 뮤텍스 잠금/해제 | 100ns |
| 주 메모리 참조 | 100ns |
| Zippy로 1KB 압축 | 10,000ns = 10µs |
| 1Gbps 네트워크로 1KB 전송 | 10,000ns = 10µs |
| SSD에서 4KB 임의 읽기 | 150,000ns = 150µs |
| 메모리에서 1MB 순차 읽기 | 250,000ns = 250µs |
| 같은 데이터센터 내 패킷 왕복 | 500,000ns = 500µs |
| SSD에서 1MB 순차 읽기 | 1,000,000ns = 1ms |
| 네트워크에서 1MB 순차 읽기 | 10,000,000ns = 10ms |
| HDD 탐색(Seek) | 10,000,000ns = 10ms |
| 패킷 왕복 (캘리포니아↔네덜란드) | 150,000,000ns = 150ms |

**핵심 교훈:**
- 메모리는 빠르지만 디스크는 아직 느리다
- 디스크 탐색은 최대한 피한다
- 단순 압축 알고리즘은 빠르다
- 데이터는 인터넷 전송 전에 압축하라
- 데이터센터 간 데이터 전송에는 비용(시간)이 크다

### 3. 가용성(Availability) 수치

| 가용률 | 하루 장애 | 한 달 장애 | 1년 장애 |
|--------|-----------|------------|----------|
| 90% (one nine) | 2.4시간 | 72시간 | 36.5일 |
| 99% (two nines) | 14.4분 | 7.31시간 | 3.65일 |
| 99.9% (three nines) | 1.44분 | 43.8분 | 8.77시간 |
| 99.99% (four nines) | 8.64초 | 4.38분 | 52.6분 |
| 99.999% (five nines) | 864ms | 26.3초 | 5.26분 |

SLA(Service Level Agreement): 서비스 제공자가 고객에게 공식 약속하는 가용성 수준. 아마존/구글/마이크로소프트는 99.9% 이상 SLA를 제공한다.

### 4. 예제: 트위터 QPS 및 저장소 추정

**가정:**
- MAU(월간 활성 사용자): 3억 명
- DAU = MAU × 50% = 1.5억 명
- 하루 평균 트윗 수: 사용자당 2건
- 트윗의 10%에 미디어(평균 1MB) 첨부
- 데이터 5년 보관

**QPS 계산:**
- 하루 총 트윗 = 1.5억 × 2 = 3억 건
- QPS = 3억 / (24 × 3,600) ≈ 3,500
- 최대(Peak) QPS ≈ 3,500 × 2 = 7,000

**저장소 계산:**
- 트윗 단위 크기: tweet_id(64B) + 텍스트(140B) = 204B
- 텍스트 저장소/일: 3억 × 204B ≈ 61.2GB
- 미디어 저장소/일: 3억 × 10% × 1MB = 30TB
- 5년 총 저장소: (61.2GB + 30TB) × 365 × 5 ≈ **54.75PB**

## 추정 팁

1. **근사치 활용:** "98,673 / 9.1"은 그냥 "100,000 / 10 = 10,000"으로 처리
2. **가정 명시:** 계산 시 가정을 적어두고 면접관과 공유
3. **단위 붙이기:** "5"가 아니라 "5MB"처럼 단위를 항상 명시
4. **자주 추정하는 것:** QPS, 최대 QPS, 저장소, 캐시 크기, 서버 수
