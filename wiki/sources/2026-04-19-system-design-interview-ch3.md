# 대규모 시스템 설계 기초 3장 — 시스템 설계 면접 공략법

- **원본:** raw/books/스크린샷 2026-04-19 오후 5.13.11~38.png (6장)
- **책:** 가상 면접 사례로 배우는 대규모 시스템 설계 기초 (Alex Xu)

## 핵심 요약

시스템 설계 면접은 이분법적 정답이 없는 개방형 문제다. 면접관은 지식의 양이 아니라 협업 능력, 트레이드오프 사고, 설계 과정의 논리성을 본다. 3장은 이 면접을 4단계 프레임워크로 체계화하고, 뉴스 피드 설계를 예제로 전체 흐름을 보여준다.

## 면접의 특성

- 정답이 없다. 설계는 열린 문제다
- 면접관은 **협업 능력**과 **트레이드오프 추론 과정**을 본다
- 무작정 설계를 시작하지 말고, 올바른 질문부터
- 잘못된 가정(assumption) 위에 세운 설계는 아무리 정교해도 실패

## 4단계 프레임워크

### 1단계: 문제 이해 및 설계 범위 확정 (15분)

면접 시작과 동시에 설계하지 말 것. 먼저 요구사항을 파악한다.

**물어봐야 할 질문 예시:**
- "구체적으로 어떤 기능을 만들어야 하나요?"
- "제품 사용자 수는 몇 명입니까?"
- "회사의 규모는 어느 정도입니까?"
- "주로 어떤 기술 스택을 사용합니까?"
- "서비스를 얼마나 빨리 확장해야 합니까?"

**뉴스 피드 예시 질문:**
- 모바일 앱과 웹 앱 둘 다 지원해야 하나요?
- 피드는 정렬 기준이 무엇인가요? (최신순? 랭킹 기반?)
- 한 사용자가 팔로우할 수 있는 친구 수 한도가 있나요?
- 트래픽 규모는 어느 정도입니까?
- 이미지, 동영상도 포함할 수 있나요?

### 2단계: 개략적 설계안 제시 및 동의 구하기 (20분)

초기 청사진(blueprint)을 만들고 면접관의 의견을 묻는다.

**포함해야 할 컴포넌트:**
- 클라이언트 (앱, 웹)
- API 서버
- 데이터 저장소
- 캐시
- CDN
- 메시지 큐

**뉴스 피드 개략 설계:**
- 피드 발행(Feed Publishing): 사용자가 포스트 작성 → 친구들의 피드에 전파
- 피드 조회(News Feed Building): 사용자가 피드 요청 → 피드 서비스가 집계해서 반환

### 3단계: 상세 설계 (15분)

개략적 설계에서 상세화가 필요한 컴포넌트를 선택하고 깊게 파고든다. 면접관과 합의해서 우선순위를 정한다.

**집중할 부분:**
- 시스템 성능 특성 (병목, 메모리 추정, 대역폭)
- URL 단축기 예: 어떻게 해시 충돌을 피하는가
- 뉴스 피드: 피드 조회 흐름, 팬아웃(fan-out) 방식 선택

**뉴스 피드 상세 컴포넌트:**
- 포스트 서비스 → 팬아웃 서비스 → 피드 캐시
- 피드 서비스 → 피드 캐시에서 조회 → 클라이언트 반환
- 알림 서비스: 팬아웃 이후 친구들에게 푸시

### 4단계: 마무리

면접 후반 추가 질문에 대응하고 마무리한다.

**할 수 있으면 좋은 것:**
- 병목 지점 지적 및 개선 방법 논의
- 운영 이슈 논의 (지표 수집, 모니터링, 로그, 배포 전략)
- 스케일업 시나리오 (5명 → 100만 명) 논의
- 미처 다루지 못한 엣지 케이스 언급

## 해야 할 것 / 하지 말아야 할 것

### 해야 할 것
- 질문을 통해 모호함을 해소하라
- 면접관이 이해한 내용을 확인받아라
- 여러 해법을 제시하고 트레이드오프를 설명하라
- 면접관과 소통하면서 진행하라
- 시간이 남으면 설계를 요약하라

### 하지 말아야 할 것
- 불분명한 상태에서 곧바로 설계 시작 ❌
- 면접관과 소통 없이 혼자 진행 ❌
- 단일 해법이 있다고 가정 ❌
- 설계가 완료됐다고 선언하고 그냥 끝내기 ❌
- 세부 사항에 집착해서 큰 그림을 놓치기 ❌

## 시간 배분 가이드 (45분 기준)

| 단계 | 활동 | 시간 |
|------|------|------|
| 1단계 | 요구사항 이해 + 설계 범위 확정 | 3~10분 |
| 2단계 | 개략적 설계 + 면접관 동의 | 10~15분 |
| 3단계 | 상세 설계 | 10~25분 |
| 4단계 | 마무리 | 3~5분 |
