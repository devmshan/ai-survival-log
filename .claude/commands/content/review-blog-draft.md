# /content:review-blog-draft — 블로그 초안 품질 리뷰

`/wiki:publish` 실행 전에 사용하는 글쓰기 품질 검증 커맨드. 구조적 lint가 잡지 못하는 서사 품질 문제를 체크합니다.

## 입력

- wiki topic 파일 경로 (예: `wiki/topics/graphify-evaluation.md`)
- 없으면 현재 작업 중인 draft를 대상으로 함

## 검증 항목

| 번호 | 항목 | 검증 질문 |
|------|------|----------|
| 1 | **헤더 품질** | 각 `##` 헤더가 해당 섹션의 통찰/발견/전환을 담고 있는가? 단순 서술 레이블인가? |
| 2 | **도입부 맥락** | 첫 2-3문단이 "이 주제가 왜 지금 중요한지"를 설명하는가? WHY가 빠진 채 바로 시작하지는 않는가? |
| 3 | **섹션별 동기** | 각 섹션이 "왜 이걸 했는가"를 명확히 하는가? 행동이나 발견의 인과가 드러나는가? |
| 4 | **기술적 깊이** | 도구/기술 평가 시 "그렇다/아니다"만이 아니라 "왜 그런지"까지 설명하는가? |
| 5 | **노이즈 필터** | 독자에게 가치 없는 구현 디테일이 있는가? (에이전트 내부 동작, 환경 세부사항, 우회 과정 등) |
| 6 | **서사 완결성** | 전체 이야기 호가 완결되는가? 맥락이 빠진 섹션이 없는가? |
| 7 | **게시물 범위** | 이 내용이 독자 입장에서 하나의 글로 읽히는가? 억지로 나뉘거나 합쳐진 건 아닌가? |

## 워크플로우

### 1단계: 기존 포스트 참조

- `output/blog/` 또는 `wiki/topics/`에서 published 포스트 1편을 읽는다.
- 톤, 헤더 스타일, 도입부 구조를 파악한다.

### 2단계: 초안 검토

- 대상 파일을 읽는다.
- 위 7개 항목을 순서대로 평가한다.
- 각 항목에 대해 **PASS / WARN / FAIL** 판정을 내린다.

### 3단계: 출력

각 항목별로:

```
[항목 번호] [항목명]: PASS / WARN / FAIL
→ 이유: (구체적)
→ 수정 제안: (WARN/FAIL일 때만)
```

### 4단계: 종합 판정

- 전체 PASS: `/wiki:publish` 진행 가능
- WARN 있음: 수정 권고 후 진행 가능
- FAIL 있음: 수정 후 재검토 필요

## 헤더 품질 판단 기준

**나쁜 헤더 (레이블):**

- "설치하고 돌려봤다" → 행동만 묘사
- "부수 효과들" → 내용을 나열할 뿐
- "시작은 Web Clipper였다" → 맥락 없는 서술

**좋은 헤더 (통찰):**

- "`pip install` 한 줄, 그리고 바로 막혔다" → 발견과 전환을 담음
- "마크다운이 이미 그 문제를 풀고 있었다" → 핵심 인사이트
- "문서만 읽으면 충돌이 보이지 않는다" → 교훈이 담긴 발견

기준: 해당 헤더만 읽어도 그 섹션에서 무엇을 얻을 수 있는지 알 수 있어야 한다.

## 노이즈 필터 기준

제거 대상:

- LLM 에이전트의 내부 실행 과정 (샌드박스 제약, 도구 선택 우회 등)
- 독자가 재현할 수 없는 일회성 환경 세부사항
- 결론에 기여하지 않는 중간 시행착오

보존 대상:

- 독자가 직접 따라할 수 있는 CLI 명령과 결과
- 결론에 직접 영향을 준 발견
- WHY를 설명하는 기술적 근거
