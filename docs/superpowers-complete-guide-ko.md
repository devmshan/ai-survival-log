## 1. 개요

Superpowers는 Claude Code를 위한 **핵심 소프트웨어 개발 워크플로우 스킬 라이브러리**입니다. ECC처럼 수백 개의 도구를 제공하는 것이 아니라, 검증된 14개의 스킬로 **개발 프로세스 자체를 체계화**합니다.

### Superpowers가 하는 일

- 코드 작성 전 브레인스토밍과 설계를 강제합니다
- TDD(테스트 주도 개발) 사이클을 엄격히 적용합니다
- 서브에이전트를 활용한 병렬 개발을 지원합니다
- 코드 리뷰와 검증을 자동화합니다
- 체계적인 디버깅 프로세스를 제공합니다

### Superpowers가 하지 않는 일

- 외부 MCP 서버를 번들하지 않습니다 (**제로 의존성**)
- 언어별 코딩 패턴이나 Rules를 제공하지 않습니다
- 보안 스캐너, Hook 자동화 등 도구를 포함하지 않습니다

> Superpowers는 **"무엇을"이 아닌 "어떻게"**에 집중합니다. 기능을 추가하는 것이 아니라, 기능을 구현하는 **프로세스**를 개선합니다.

## 3. 핵심 철학

Superpowers는 4가지 핵심 원칙을 기반으로 설계되었습니다:

### 3.1 TDD (테스트 주도 개발)

```
실패하는 테스트 없이 프로덕션 코드를 작성하지 않는다.
```

이것은 선택사항이 아닌 **절대 규칙(Iron Law)**입니다. 테스트 없이 코드를 먼저 작성했다면, 그 코드를 **삭제**해야 합니다 — 참고용으로도, 수정용으로도 남기지 않습니다.

### 3.2 체계적 접근 (Systematic over Ad-hoc)

```
추측하지 않는다. 조사하고, 분석하고, 검증한다.
```

디버깅할 때 "이걸 바꿔볼까?"가 아니라 "원인이 무엇인가?"를 먼저 묻습니다. 3번 이상 수정이 실패하면 멈추고 아키텍처를 재검토합니다.

### 3.3 복잡도 감소 (Complexity Reduction)

```
단순함이 기본이다. 복잡함은 증명이 필요하다.
```

DRY, YAGNI 원칙을 엄격히 적용합니다. 불필요한 추상화, 플레이스홀더, "나중에" 구현을 허용하지 않습니다.

### 3.4 증거 기반 (Evidence over Claims)

```
"완료했습니다"가 아니라 "테스트가 통과했습니다"로 증명한다.
```

작업 완료를 선언하기 전에 반드시 검증 명령을 실행하고, 그 출력을 확인합니다.

## 4. 전체 워크플로우 흐름

Superpowers는 소프트웨어 개발의 전체 라이프사이클을 커버하는 워크플로우를 제공합니다:

```
┌─────────────────────────────────────────────────────┐
│                    세션 시작                          │
│              using-superpowers 자동 로드              │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  1. brainstorming (브레인스토밍)                      │
│     아이디어 → 소크라틱 대화 → 설계 문서              │
│     ⚠️ 설계 승인 없이 코드 작성 금지                  │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  2. using-git-worktrees (워크트리 생성)              │
│     격리된 작업 환경 세팅                             │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  3. writing-plans (계획 수립)                         │
│     설계 → 2-5분 단위 태스크로 분해                   │
│     각 태스크에 정확한 파일 경로, 코드, 검증 명령      │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  4. 실행 (택 1)                                      │
│     A. subagent-driven-development (서브에이전트)     │
│        태스크당 1개 서브에이전트 → 2단계 리뷰         │
│     B. executing-plans (순차 실행)                    │
│        태스크를 순서대로 직접 실행                     │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  5. test-driven-development (각 태스크 내부)         │
│     RED → GREEN → REFACTOR 사이클 반복               │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  6. requesting-code-review (코드 리뷰)               │
│     code-reviewer 서브에이전트 자동 디스패치          │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  7. verification-before-completion (최종 검증)       │
│     증거 없이 완료 선언 금지                          │
└──────────────────────┬──────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────┐
│  8. finishing-a-development-branch (브랜치 마무리)    │
│     머지 / PR / 유지 / 폐기 중 선택                  │
└─────────────────────────────────────────────────────┘
```

**버그 발생 시:**

```
systematic-debugging 스킬 자동 활성화
→ 원인 조사 → 패턴 분석 → 가설 검증 → 구현
```

## 5. 스킬 상세 가이드

### 5.1 using-superpowers (부트스트랩)

**분류:** 메타/부트스트랩
**자동 호출:** 세션 시작 시 SessionStart Hook으로 자동 로드

모든 대화 시작 시 자동으로 로드되는 부트스트랩 스킬입니다. 다른 모든 스킬의 호출을 관리하며, **"관련 스킬이 1%라도 적용될 가능성이 있으면 반드시 호출하라"**는 규칙을 세웁니다.

**핵심 규칙:**
- 스킬 체크가 모든 응답보다 먼저 와야 합니다
- 명확화 질문을 하기 전에도 스킬을 먼저 확인합니다
- 프로세스 스킬(brainstorming, debugging)이 구현 스킬보다 먼저 적용됩니다

**합리화 방지 표 (Red Flags):**

| 이런 생각이 들면 | 현실 |
|-----------------|------|
| "간단한 질문인데" | 질문도 태스크입니다. 스킬을 확인하세요. |
| "먼저 코드를 좀 볼게요" | 스킬이 탐색 방법을 알려줍니다. |
| "이건 스킬까지 필요 없어" | 스킬이 있으면 사용하세요. |
| "먼저 하나만 해볼게요" | 확인 먼저, 행동은 그 다음. |

---

### 5.2 brainstorming (브레인스토밍)

**분류:** 협업/설계
**호출 시점:** 새로운 기능, 컴포넌트, 수정 등 모든 창의적 작업 전

**절대 규칙:** 설계가 승인되기 전에 코드를 작성하지 않습니다.

**9단계 체크리스트:**

1. 컨텍스트 탐색 — 관련 코드와 패턴 조사
2. 비주얼 컴패니언 제안 — 브라우저 기반 목업/다이어그램 (선택)
3. 명확화 질문 — 한 번에 하나, 객관식 선호
4. 2-3가지 접근 방식 제안 — 트레이드오프 포함
5. 선택된 접근 방식 상세 설계
6. 설계 문서 작성
7. 자체 검토
8. 사용자 검토 요청
9. writing-plans로 전환

**사용 예시:**

```
사용자: "사용자 인증을 추가해줘"

Claude: (brainstorming 스킬 활성화)

"인증 시스템을 설계하기 전에 몇 가지 확인하겠습니다.

현재 프로젝트에서 세션 관리를 어떻게 하고 있나요?
A) 서버 사이드 세션 (express-session 등)
B) JWT 토큰 (stateless)
C) 아직 없음 — 새로 구축

소셜 로그인은 필요한가요?
A) Google만
B) Google + GitHub
C) 불필요 — 이메일/비밀번호만
D) 나중에 추가 가능하도록 설계"
```

**비주얼 컴패니언:**

브레인스토밍 단계에서 비주얼 컴패니언을 시작할 수 있습니다. Node.js 기반 로컬 서버가 브라우저에서 실시간 목업과 다이어그램을 렌더링합니다.

### 5.3 writing-plans (계획 수립)

**분류:** 협업/계획
**호출 시점:** 스펙이나 요구사항이 있고, 코드를 작성하기 전

설계 문서를 **2-5분 단위의 소형 태스크**로 분해합니다. 각 태스크에는 다음이 반드시 포함됩니다:

- 정확한 파일 경로
- 완전한 코드 (플레이스홀더 금지)
- 검증 명령어
- 커밋 지점

**작성 원칙:**
- DRY, YAGNI 엄격 적용
- TDD — 테스트가 먼저
- 작고 자주 커밋
- 기존 패턴 재사용 우선

**태스크 예시:**

```markdown
### Task 3: UserService 테스트 작성

**파일:** src/__tests__/user-service.test.ts

테스트 케이스:
- createUser가 유효한 이메일로 사용자를 생성하는지 확인
- createUser가 중복 이메일에 에러를 던지는지 확인
- getUser가 존재하지 않는 ID에 null을 반환하는지 확인

**검증:** npm test -- --grep "UserService"
**커밋:** "test: add UserService unit tests"
```

### 5.4 subagent-driven-development (서브에이전트 개발)

**분류:** 협업/실행
**호출 시점:** 구현 계획이 있고, 독립적인 태스크를 병렬로 실행할 때

각 태스크마다 **독립적인 서브에이전트를 디스패치**하여 구현합니다. 서브에이전트 완료 후 **2단계 리뷰**가 자동으로 진행됩니다:

1. **스펙 준수 리뷰** — 요구사항 충족 확인
2. **코드 품질 리뷰** — 코드 표준 준수 확인

**서브에이전트 상태:**

| 상태 | 의미 | 대응 |
|------|------|------|
| `DONE` | 완료 | 다음 태스크 진행 |
| `DONE_WITH_CONCERNS` | 완료 (우려사항 있음) | 우려사항 검토 후 진행 |
| `NEEDS_CONTEXT` | 추가 컨텍스트 필요 | 정보 제공 후 재실행 |
| `BLOCKED` | 차단됨 | 차단 원인 해결 |

**모델 선택 가이드:**
- **기계적 작업** (테스트 추가, 단순 구현) → 저렴한 모델 (Haiku)
- **아키텍처 결정** (설계, 리팩토링) → 고성능 모델 (Opus)

### 5.5 executing-plans (순차 실행)

**분류:** 협업/실행
**호출 시점:** 서브에이전트 없이 계획을 직접 실행할 때

subagent-driven-development의 단순화 버전입니다. 태스크를 순서대로 직접 실행하며, 차단(blocker) 발생 시 추측하지 않고 멈춥니다.

> **권장:** 서브에이전트를 사용할 수 있다면 `subagent-driven-development`를 권장합니다.

---

### 5.6 dispatching-parallel-agents (병렬 에이전트)

**분류:** 협업/실행
**호출 시점:** 2개 이상의 독립적인 작업을 동시에 처리할 때

**사용 조건:**
- 태스크가 서로 독립적 (공유 상태 없음)
- 순차적 의존성 없음
- 각 태스크가 다른 문제 도메인에 속함

**에이전트 프롬프트 구조:**
```
- 집중 범위 (Focused Scope)
- 명확한 목표 (Clear Goal)
- 제약사항 (Constraints)
- 기대 출력 (Expected Output)
```

**사용 예시:**

```
"API 서버 리팩토링과 프론트엔드 테스트를 동시에 진행합니다.

에이전트 1: API 라우트 리팩토링
에이전트 2: React 컴포넌트 테스트 작성

두 에이전트의 결과를 통합합니다."
```

### 5.7 test-driven-development (TDD)

**분류:** 테스팅
**호출 시점:** 모든 기능 구현 및 버그 수정 시

**절대 규칙:** 실패하는 테스트 없이 프로덕션 코드를 작성하지 않습니다.

**RED-GREEN-REFACTOR 사이클:**

```
RED    → 실패하는 테스트 작성
GREEN  → 테스트를 통과하는 최소한의 코드 작성
REFACTOR → 코드 정리 (테스트는 여전히 통과)
반복
```

**위반 시 처리:**
코드를 테스트보다 먼저 작성했다면 → **삭제합니다** (예외 없음)
- 참고용으로 남기지 않습니다
- 수정해서 재사용하지 않습니다
- 테스트부터 다시 시작합니다

**합리화 방지 표:**

| 이런 생각이 들면 | 현실 |
|-----------------|------|
| "테스트할 수 없는 코드인데" | 테스트 가능하게 설계를 바꾸세요 |
| "빠르게 프로토타입만" | 프로토타입에도 테스트가 필요합니다 |
| "명확한 코드라 테스트 불필요" | 모든 코드에 테스트가 필요합니다 |
| "리팩토링만 하는 건데" | 리팩토링 전 기존 테스트 확인이 먼저입니다 |
| "프로덕션 코드 참고해서 테스트 작성" | 코드가 테스트를 따르지, 반대가 아닙니다 |

**포함된 참고 자료:**
- `testing-anti-patterns.md` — 흔한 테스트 안티패턴 가이드

---

### 5.8 systematic-debugging (체계적 디버깅)

**분류:** 디버깅
**호출 시점:** 버그, 테스트 실패, 예상치 못한 동작 발견 시

**절대 규칙:** 원인 조사 없이 수정을 시도하지 않습니다.

**4단계 프로세스:**

```
Phase 1: 원인 조사 (Root Cause Investigation)
→ 콜스택 역추적, 로그 분석, 상태 검사

Phase 2: 패턴 분석 (Pattern Analysis)
→ 유사 이슈, 재현 조건, 영향 범위 파악

Phase 3: 가설 및 검증 (Hypothesis and Testing)
→ 가설 수립 → 실험 → 확인 또는 기각

Phase 4: 구현 (Implementation)
→ 확인된 원인에 기반한 수정
```

**3번 실패 규칙:** 3번 이상 수정이 실패하면 **즉시 멈추고** 아키텍처를 재검토합니다.

**포함된 참고 자료:**
- `root-cause-tracing.md` — 콜스택 역추적 방법론
- `defense-in-depth.md` — 다중 방어 레이어 검증
- `condition-based-waiting.md` — 타임아웃 대신 조건 기반 대기
- `find-polluter.sh` — 테스트 오염 탐지 스크립트

---

### 5.9 verification-before-completion (완료 전 검증)

**분류:** 디버깅/검증
**호출 시점:** 작업 완료 선언, 커밋, PR 생성 직전

**절대 규칙:** 새로운 검증 증거 없이 완료를 선언하지 않습니다.

**게이트 함수:**

```
1. 검증 명령어 식별 (npm test, cargo test, go test 등)
2. 명령어 실행
3. 전체 출력 읽기
4. 출력이 성공을 확인하는지 검증
5. 그 다음에만 완료 선언
```

**적용 대상:**
- 테스트 통과 여부
- 빌드 성공 여부
- 요구사항 충족 여부
- 서브에이전트 작업 결과

---

### 5.10 requesting-code-review (코드 리뷰 요청)

**분류:** 협업/리뷰
**호출 시점:** 태스크 완료 후, 주요 기능 구현 후, 머지 전

`superpowers:code-reviewer` 서브에이전트를 디스패치하여 코드 리뷰를 수행합니다.

**리뷰 범위:**
- 계획 준수 분석
- 코드 품질 평가
- 아키텍처 및 설계 리뷰
- 문서 및 표준 준수
- 이슈 식별 (Critical / Important / Suggestion)

**사용 예시:**

```
"Task 5 구현을 완료했습니다. 코드 리뷰를 요청합니다."

→ Claude가 code-reviewer 서브에이전트에게:
  - 구현한 내용 (WHAT_WAS_IMPLEMENTED)
  - 계획/요구사항 (PLAN_OR_REQUIREMENTS)
  - 커밋 범위 (BASE_SHA...HEAD_SHA)
  를 전달하고 리뷰 결과를 받습니다.
```

---

### 5.11 receiving-code-review (코드 리뷰 수신)

**분류:** 협업/리뷰
**호출 시점:** 코드 리뷰 피드백을 받았을 때

**핵심 원칙:** 기술적 평가가 먼저, 감정적 반응은 불필요합니다.

**금지 표현:**
- "You're absolutely right!" ❌
- "Great point!" ❌
- "Oh, I should have caught that!" ❌

**올바른 대응 프로세스:**

```
READ → UNDERSTAND → VERIFY → EVALUATE → RESPOND → IMPLEMENT
```

1. **읽기** — 피드백 전체를 먼저 읽습니다
2. **이해** — 무엇을 지적하는지 파악합니다
3. **검증** — 지적 사항이 실제로 맞는지 코드로 확인합니다
4. **평가** — 동의하면 수정, 동의하지 않으면 기술적 근거를 들어 반박합니다
5. **응답** — 불명확한 항목은 전부 명확히 한 후에만 구현합니다
6. **구현** — 합의된 수정만 적용합니다

---

### 5.12 using-git-worktrees (Git Worktree)

**분류:** 협업/환경
**호출 시점:** 기능 개발 시작 시, 현재 작업 공간과 격리가 필요할 때

Git Worktree를 생성하여 격리된 작업 환경을 만듭니다.

**디렉토리 선택 순서:**
1. 기존 `.worktrees/` 디렉토리
2. 기존 `worktrees/` 디렉토리
3. CLAUDE.md에 명시된 경로
4. 사용자에게 질문

**안전 규칙:**
- 워크트리 디렉토리가 `.gitignore`에 포함되어 있는지 반드시 확인
- 프로젝트 세팅 자동 감지 (npm install / cargo build / pip install / go mod download)
- 클린 테스트 베이스라인 확인

---

### 5.13 finishing-a-development-branch (브랜치 마무리)

**분류:** 협업/통합
**호출 시점:** 구현 완료, 모든 테스트 통과 후

**정확히 4가지 선택지를 제시합니다:**

| 옵션 | 설명 |
|------|------|
| 1. 로컬 머지 | 베이스 브랜치에 로컬로 머지 |
| 2. Push + PR | 원격에 Push하고 PR 생성 |
| 3. 유지 | 브랜치를 그대로 유지 |
| 4. 폐기 | 브랜치 삭제 (타이핑 확인 "discard" 필요) |

옵션 1과 4 선택 시 워크트리가 자동으로 정리됩니다.

---

### 5.14 writing-skills (스킬 작성)

**분류:** 메타
**호출 시점:** 새 스킬을 만들거나 기존 스킬을 수정할 때

스킬 작성에도 TDD를 적용합니다:

```
RED    → 서브에이전트로 베이스라인 측정 (스킬 없이)
GREEN  → 스킬 작성
REFACTOR → 허점 보완, 토큰 효율 최적화
```

**포함된 참고 자료:**
- `anthropic-best-practices.md` — Anthropic 공식 스킬 작성 모범 사례 (45KB)
- `persuasion-principles.md` — 스킬 방어 강화를 위한 설득 원칙
- `testing-skills-with-subagents.md` — 서브에이전트를 활용한 스킬 테스팅

## 7. 실전 워크플로우

### 7.1 새 기능 개발 (전체 흐름)

가장 일반적인 사용 시나리오입니다.

```
사용자: "댓글 기능을 추가해줘"

Step 1: 브레인스토밍 (자동)
→ brainstorming 스킬 활성화
→ "댓글에 대댓글은 필요한가요? 멘션 기능은?"
→ 설계 문서 작성 및 승인

Step 2: 계획 수립
→ writing-plans 스킬 활성화
→ 10개의 2-5분 태스크로 분해
→ 각 태스크에 파일 경로, 코드, 검증 명령 포함

Step 3: 워크트리 생성 (선택)
→ using-git-worktrees로 격리 환경 생성

Step 4: 실행
→ subagent-driven-development 활성화
→ 태스크 1: Comment 모델 테스트 작성 (TDD RED)
→ 태스크 2: Comment 모델 구현 (TDD GREEN)
→ 태스크 3: API 라우트 테스트 작성
→ ... 각 태스크 후 2단계 리뷰

Step 5: 코드 리뷰
→ requesting-code-review → code-reviewer 에이전트 디스패치

Step 6: 검증
→ verification-before-completion → 전체 테스트 실행

Step 7: 브랜치 마무리
→ finishing-a-development-branch → PR 생성
```

### 7.2 버그 수정

```

→ systematic-debugging 스킬 활성화

Phase 1: 원인 조사
→ 에러 로그 분석 → 콜스택 추적
→ "세션 토큰 만료 처리에서 race condition 발견"

Phase 2: 패턴 분석
→ "토큰 갱신과 요청이 동시에 발생할 때만 재현"

Phase 3: 가설 검증
→ "atomic 토큰 갱신으로 해결 가능" → 테스트로 검증

Step 2: TDD 수정
→ test-driven-development 활성화
→ RED: race condition 재현 테스트 작성
→ GREEN: atomic 토큰 갱신 구현
→ REFACTOR: 정리

Step 3: 검증 및 완료
→ verification-before-completion → 테스트 통과 확인
```

## 8. Hook 시스템

Superpowers는 **단 1개의 Hook**만 사용합니다:

### SessionStart Hook

| 항목 | 내용 |
|------|------|
| **이벤트** | `SessionStart` (시작, /clear, compact 시) |
| **동작** | `using-superpowers` SKILL.md 내용을 세션 컨텍스트에 주입 |
| **비동기** | `false` (완료까지 대기) |

**동작 과정:**

```
세션 시작
    ↓
hooks/session-start 스크립트 실행
    ↓
using-superpowers SKILL.md 읽기
    ↓
JSON으로 포맷 (additionalContext)
    ↓
<EXTREMELY_IMPORTANT> 태그로 감싸서 주입
    ↓
이후 모든 대화에서 스킬 체크가 자동 적용
```

레거시 스킬 디렉토리(`~/.config/superpowers/skills`)가 존재하면 마이그레이션 경고를 표시합니다.

## 11. 문제 해결 (FAQ)

### Q: 스킬이 자동으로 적용되지 않습니다.

**원인:** SessionStart Hook이 실행되지 않았을 수 있습니다.
**해결:**
1. 새 세션을 시작하거나 `/clear`를 실행하세요
2. 플러그인이 올바르게 설치되었는지 확인하세요:
   ```
   /plugin list
   ```

---

### Q: "설계부터 하자"는 규칙이 너무 엄격합니다.

**설명:** Superpowers의 brainstorming은 **"설계가 승인될 때까지 코드 금지"**가 기본입니다. 그러나 사용자 지시가 항상 최우선입니다.
**해결:** "브레인스토밍 건너뛰고 바로 구현해줘"라고 지시하면 됩니다. 사용자 명시적 지시 > Superpowers 스킬 > 기본 시스템 프롬프트 순서로 우선됩니다.

---

### Q: TDD가 필요 없는 간단한 작업에도 테스트를 강제합니다.

**해결:** "이건 테스트 없이 진행해줘"라고 명시적으로 지시하세요. Superpowers는 사용자 지시를 최우선으로 따릅니다.

---

### Q: 레거시 스킬 경고가 표시됩니다.

**원인:** `~/.config/superpowers/skills/` 디렉토리에 이전 버전의 스킬이 남아있습니다.
**해결:**
```bash
rm -rf ~/.config/superpowers/skills/
```

---

### Q: 비주얼 컴패니언이 동작하지 않습니다.

**원인:** Node.js 서버 실행 문제일 수 있습니다.
**해결:** 브레인스토밍 시 비주얼 컴패니언은 선택 사항입니다. "비주얼 없이 텍스트로만 진행해줘"라고 하면 됩니다.

---

### Q: Superpowers와 ECC가 충돌합니다.

**해결:** 일반적으로 충돌하지 않습니다. 두 플러그인은 서로 다른 영역을 담당합니다. 만약 동일한 상황에서 두 스킬이 경쟁하면, 명시적으로 사용할 스킬을 지정하세요:
```
"/superpowers:brainstorming으로 설계를 진행해줘"
"/ecc:plan으로 계획을 세워줘"
```
