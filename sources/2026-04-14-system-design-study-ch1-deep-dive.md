# 시스템 설계 스터디 — 1장 심화 대화 (2026-04-14)

> 출처: Claude와의 스터디 대화
> 주제: 캐시 전략 심화, Memcached API, SAML, 샤딩, 메시지 큐

## 주요 내용

### Memcached API
- set(덮어씀), add(없을 때만), get, delete, incr/decr
- Read-Through에서 add 사용 → 레이스 컨디션 방지
- Write-Through에서 set 사용 → 항상 최신값 갱신

### 캐시 전략 심화
- Read-Through: 읽기 요청 + 캐시 미스 시 DB에서 읽어 캐시 채움
- Write-Through: 쓰기 요청 시 캐시 + DB 동시 업데이트
- Write-Back: 캐시에만 먼저 쓰고 DB는 나중에 비동기로 저장
- 두 전략은 상호 배타적이지 않음 (함께 사용 가능)

### 캐시 계층
- 브라우저 캐시: 내 컴퓨터, 나만 사용, 제어권 클라이언트
- CDN: 네트워크 엣지, 모든 유저 공유, 정적 콘텐츠 전용
- 인메모리 캐시(Redis/Memcached): 서버 제어, 동적 데이터

### 캐시 일관성 유지 전략
- TTL: 단순하지만 시간차 허용
- 명시적 무효화: DB 업데이트 후 즉시 캐시 delete
- 이벤트 기반(CDC): 메시지 큐로 자동 무효화
- 버전 키: 키 변경으로 구버전 자연 소멸

### SAML 프로토콜
- IdP(인증 담당) + SP(서비스) 구조
- 세션이 두 개 생성: IdP 세션 + SP 세션
- SSO: IdP 세션 살아있으면 다른 SP 자동 인증
- SLO(글로벌 로그아웃): 모든 SP에 로그아웃 전파
- 전자결재 XML 오류 원인 분석: 세션 만료 불일치, Assertion 만료, 클럭 스큐, XML 직렬화 버그

### DB 다중화 동기화
- 동기식: 항상 일치 보장, 느림, 블로킹 위험
- 비동기식: 빠름, 복제 지연(Replication Lag) 발생
- 메커니즘: 바이너리 로그(binlog) 구독
- 해결: Read-Your-Writes (방금 쓴 유저는 주 DB에서 읽기)

### 샤딩 심화
- 수평 샤딩(Horizontal): 레코드 단위 분리 (가장 일반적)
- 수직 샤딩(Vertical): 테이블/도메인 단위 분리
- 샤딩 방법: 범위 기반, 해시 기반(일반적), 위치 기반(글로벌)
- 버티컬 샤딩 후 조인 문제: 애플리케이션 레벨 조인, 데이터 중복, 데이터 웨어하우스

### 메시지 큐 심화
- 소비자 = 큐에서 메시지 꺼내 처리하는 서버/프로세스
- 경쟁 소비자 패턴: 소비자 여러 개로 병렬 처리
- basicQos(1): 한 번에 1개만 처리, 과부하 방지
- ack/nack: 소비자 죽어도 메시지 유실 없음

### 1장 종합 정리
- SPOF 제거가 다중화의 목표
- 독립적 서비스 분할: 장애 격리 + 독립적 스케일 아웃 + 독립적 배포
- 다중화(가용성) + 독립 분할(확장성 + 장애 격리)
